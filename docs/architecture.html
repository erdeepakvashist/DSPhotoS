Background & Lifecycle

Scan thread

  • Runs as a daemon thread; a ThreadPoolExecutor (cpu-2, capped 2–6 workers) handles the CPU-bound per-photo work
  • Worker threads run at BELOW_NORMAL priority on Windows to keep the UI responsive during inference
  • scanner.STATUS is polled by the frontend every tick and drives the top-bar scan chip plus the Settings progress bar
  • Stopping sets a threading.Event; in-flight photos finish, queued ones are cancelled, partial progress is kept

App lifecycle

  • POST /api/app/restart spawns a detached relaunch, then exits via a delayed os._exit(0) so the response flushes first
  • POST /api/app/stop does the same without relaunching
  • Both routes — like every other state-changing endpoint — are rejected if the request's Origin/Referer isn't this machine
